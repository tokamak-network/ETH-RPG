## 1. 전략 방향 요약

### 핵심 목표
ETH-RPG를 **라이트 온체인 게임**으로 발전시켜 Tokamak Network L2의 활성화(브릿징 볼륨 + TVL 증가)에 기여한다.

### 확정된 결정
| 항목 | 결정 | 근거 |
|------|------|------|
| 게임 범위 | 라이트 게임 (PvP, 퀘스트, 업적) | 기존 캐릭터 카드 활용, 개발 리소스 고려 |
| 통화 모델 | 이중 통화 (TON + SOUL) | 가격 변동/규제 리스크 최소화 |
| L2 플랫폼 | Tokamak Network (Titan 또는 앱롤럽) | 킬러앱 포지셔닝 가능, 그랜트 기회 |
| 파트너십 | 독립 개발 후 Tokamak에 제안 | 작동하는 제품 + 유저 데이터로 설득 |
| 온보딩 | 3단계 점진적 전환 | 진입장벽 제로 유지 |

### 미결정 사항
| 항목 | 옵션 | 결정 시기 |
|------|------|-----------|
| L2 선택 | Titan L2 vs 전용 앱롤럽 | Tokamak 팀 상의 후 |
| SOUL↔TON 교환 방향 | 단방향 vs 제한적 양방향 | 법률 검토 후 |
| 시즌 패스 유료화 | 무료 only vs 프리미엄 트랙 | L2 도입 후 |

### 1-1. 시장 분석 & 경쟁 환경

#### 온체인 프로필/게이미피케이션 시장

온체인 활동을 시각화하거나 게임화하는 서비스는 이미 다수 존재한다. Eth·RPG의 차별화를 위해 기존 서비스를 분석한다.

#### 경쟁 서비스 비교

| 항목 | DegenScore | Layer3 / Rabbithole | Galxe | Eth·RPG |
|------|-----------|-------------------|-------|---------|
| **카테고리** | 온체인 신용 점수 | 퀘스트 플랫폼 | 캠페인 플랫폼 | 온체인 RPG 프로필 |
| **핵심 기능** | 지갑 점수 산출 (0-1000) | 온체인 퀘스트 수행 → 보상 | 프로젝트 캠페인 참여 → 보상 | 지갑 분석 → RPG 캐릭터 생성 |
| **수익 모델** | SBT 민팅, B2B API | B2B (프로젝트 비용), 토큰 | B2B (캠페인 수수료) | 프리미엄 스킨, 시즌 패스 (계획) |
| **유저 동기** | 온체인 신용 증명 | 에어드랍/보상 획득 | 에어드랍/보상 획득 | 자기 표현 + PvP 재미 |
| **진입 장벽** | 지갑 연결 필수 | 지갑 연결 + 서명 | 지갑 연결 + 서명 | **주소 입력만 (연결 불필요)** |
| **리텐션 동기** | 점수 변화 추적 | 새 퀘스트/에어드랍 | 새 캠페인 | PvP + 시즌 랭킹 + 퀘스트 |
| **MAU 추정** | ~50K-100K (2024 피크) | ~200K-500K | ~1M+ | 미출시 (목표 1K-10K) |
| **공유 바이럴성** | 낮음 (숫자만) | 중간 (퀘스트 완료 공유) | 중간 (배지 공유) | **높음 (RPG 카드 = 시각적 콘텐츠)** |

#### Eth·RPG의 차별화 포인트

1. **진입 장벽 제로**: 지갑 연결 없이 주소만 입력. 경쟁 서비스 대비 유일하게 서명 불필요
2. **시각적 공유성**: RPG 캐릭터 카드는 "숫자"나 "배지"보다 SNS에서 시각적 임팩트가 크다
3. **자기 표현 동기**: 보상(에어드랍) 동기가 아닌 "내 지갑의 정체성" 동기 -- 보상 시즌 종료 시에도 유저 동기 유지
4. **PvP 소셜 요소**: "내 지갑 vs 네 지갑" 배틀은 타 서비스에 없는 고유 기능

#### 지속 가능한 경쟁 우위 분석

| 차별화 요소 | 방어 가능성 | 근거 |
|------------|-----------|------|
| RPG 캐릭터 카드 포맷 | 중간 | 비주얼은 모방 가능하나, 8클래스+AI 로어+전투 시스템의 복합적 깊이가 장벽 |
| 진입 장벽 제로 | 낮음 | 기술적으로 쉽게 복제 가능 |
| PvP 배틀 시스템 | 높음 | 결정론적 시뮬레이션 + 클래스 상성 + 스킬 시스템의 복합 설계 |
| AI 히어로 로어 | 중간 | LLM 기반이라 복제 가능하나, 프롬프트 품질과 폴백 시스템이 차별점 |
| Tokamak L2 통합 | 높음 | 생태계 최초 B2C 앱 포지셔닝, 그랜트/파트너십을 통한 선점 |

## 2. 이중 통화 모델 (TON + SOUL)

### 구조도
```
                    ┌────────────────────────────────────┐
                    │         TOKAMAK L2 (온체인)          │
                    │                                     │
  L1 TON ─브릿지─> │  TON 예치 ─교환─> SOUL (게임 내 토큰) │
                    │    |                  |              │
                    │    |                  v              │
                    │    |         ┌──────────────┐        │
                    │    |         │  게임 경제 순환 │        │
                    │    |         │              │        │
                    │    |         │ PvP 입장료    │        │
                    │    |         │ 장비 강화     │─> 소각  │
                    │    |         │ 스킬 해금     │        │
                    │    |         │ 시즌 패스     │        │
                    │    |         │              │        │
                    │    |         │ PvP 보상     │        │
                    │    |         │ 퀘스트 보상   │─> 유저  │
                    │    |         │ 업적 보상     │        │
                    │    |         │ 시즌 랭킹 보상│        │
                    │    |         └──────────────┘        │
                    │    |                                 │
                    │    └─> TON 풀 (보상 재원 + 운영비)     │
                    └────────────────────────────────────┘
```

### TON의 역할
- **진입 화폐**: L1→L2 브릿징의 매개체
- **가치 저장**: 게임 참여를 위한 예치
- **Tokamak 기여**: 브릿징 볼륨 → TVL 증가

### SOUL의 역할
- **게임 화폐**: 모든 게임 내 거래/보상의 단위
- **경제 안정**: 고정 교환비로 게임 내 가격 안정
- **규제 회피**: 게임 밖 환전 불가(단방향) 시 사행성 분류 회피

### 교환 메커니즘
```
TON → SOUL: 가능 (1 TON = 1,000 SOUL, 고정 또는 오라클 기반, 일일 상한 5,000 SOUL/인)
SOUL → TON: 불가 (단방향) — 규제 안전성 우선
         또는 제한적 가능 — 법률 검토 후 결정
```

### SOUL 소비 경로 (Sink)
| 사용처 | SOUL 비용 | 빈도 | 소각 비율 |
|--------|----------|------|----------|
| 랭킹 PvP 입장료 | 50 SOUL | 매 배틀 | 30% 소각, 70% 보상풀 |
| 스킬 해금 (2번째 스킬) | 500 SOUL | 1회성 | 100% 소각 |
| 장비 강화 | 100~500 SOUL | 반복 | 50% 소각, 실패 시 100% |
| 카드 스킨 변경 | 200 SOUL | 1회성 | 100% 소각 |
| 시즌 패스 구매 | 2,000 SOUL | 시즌당 1회 | 50% 소각, 50% 보상풀 |
| 토너먼트 참가비 | 500 SOUL | 이벤트당 | 20% 소각, 80% 상금풀 |

### SOUL 획득 경로 (Faucet)
| 획득처 | SOUL 보상 | 일일 상한 |
|--------|----------|----------|
| TON → SOUL 교환 | 1 TON = 1,000 SOUL | 5,000 SOUL/일 |
| 랭킹 PvP 승리 | 30~70 SOUL | 300 SOUL/일 |
| 일일 미션 (3개) | 각 20 SOUL | 60 SOUL/일 |
| 주간 퀘스트 | 200 SOUL | 200 SOUL/주 |
| 업적 달성 | 50~500 SOUL | 1회성 |
| 시즌 랭킹 보상 | 1,000~10,000 SOUL | 시즌당 1회 |

### 인플레이션 억제
- 보상에 로그 스케일 적용 (기존 stats.ts와 동일 패턴)
- 일일/주간 보상 하드캡
- PvP 입장료 소각 (30%)
- 장비 강화 실패 소각 (100%)
- 시즌 리셋 시 미사용 SOUL 15% 자동 소각
- 프리미엄 스킨/이펙트 구매 = 순수 소각

### 2-1. SOUL 경제 시뮬레이션

#### 유저 1인당 일일 SOUL 흐름 (활성 유저 기준)

**획득 (Faucet):**
| 활동 | SOUL/일 | 참여율 가정 |
|------|---------|------------|
| PvP 승리 (3승/6배틀) | 150 | 100% (활성 유저) |
| 일일 미션 3개 완료 | 60 | 70% |
| 주간 퀘스트 (1/7) | 29 | 50% |
| **소계** | **~210 SOUL/일** | |

**소비 (Sink):**
| 활동 | SOUL/일 | 참여율 가정 |
|------|---------|------------|
| PvP 입장료 (50 x 6) | 300 | 100% |
| 장비 강화 시도 (1회) | 200 | 30% |
| **소계** | **~360 SOUL/일** | |

**순 흐름**: 활성 유저는 일일 약 -150 SOUL (소비 > 획득)
-> 활성 유저는 TON->SOUL 교환을 통해 SOUL을 충전해야 함
-> 이는 곧 **Tokamak L2 브릿징 수요**로 직결

#### 6개월 시뮬레이션 (시나리오별)

**변수 정의:**
- 신규 유저 SOUL 보너스: Welcome Chest 500 SOUL
- 시즌 리셋 소각: 미사용 SOUL 15%
- PvP 입장료 소각율: 30%
- TON->SOUL 교환: 활성 유저 주 1회, 500 SOUL 평균

| 지표 | 비관 (DAU 30) | 중립 (DAU 100) | 낙관 (DAU 500) |
|------|--------------|---------------|---------------|
| 월간 신규 유저 | 50 | 200 | 1,000 |
| 월간 SOUL 발행 (게임 보상) | 189,000 | 630,000 | 3,150,000 |
| 월간 TON→SOUL 교환 유입 | 90,000 | 300,000 | 1,500,000 |
| 월간 Welcome Chest (신규 유저) | 25,000 | 100,000 | 500,000 |
| 월간 SOUL 소각 (입장료+강화+스킨) | 226,800 | 756,000 | 3,780,000 |
| 월간 순 SOUL 흐름 (보상+교환+Welcome-소각) | +77,200 | +274,000 | +1,370,000 |
| 6개월 후 총 SOUL 공급량 | ~1.2M | ~4.0M | ~20.0M |
| 6개월 후 인당 보유량 | ~2,000 | ~2,000 | ~2,000 |

*주: TON→SOUL 교환 = 활성 유저 주 1회 평균 500 SOUL (DAU × 500 × 30/7). Welcome Chest = 신규 유저 × 500 SOUL.*

**핵심 발견:**
1. **게임 보상만으로는 디플레이션**: 활성 유저 기준 게임 내 순 소비가 양수 (소비 360 > 획득 210)
2. **TON 교환 + Welcome Chest가 순 공급 증가의 원인**: 외부 유입이 게임 내 소각을 상쇄
3. **교환 상한이 핵심 제어 변수**: 일일 TON→SOUL 교환 상한 5,000 SOUL/인으로 인플레이션 속도 제한

#### 민감도 분석

| 변수 변화 | SOUL 공급 영향 | 조치 |
|----------|--------------|------|
| 소각율 30%->20% | 공급 +15% | PvP 입장료 인상 또는 소각율 복원 |
| 일일 미션 보상 2배 | 공급 +28% | 미션 보상 상한 강화 |
| TON->SOUL 무제한 | 공급 제어 불가 | 반드시 일일/주간 교환 상한 설정 |
| DAU 10배 급증 | 소각도 10배 -> 비율 유지 | Sink/Faucet 비율이 DAU 독립적 |

#### 설계 결론
- 현재 Sink/Faucet 비율은 소비 우위로 설계되어 있어 인플레이션 리스크 낮음
- **단, TON->SOUL 교환에 반드시 일일 상한 설정 필요** (현재 "무제한"은 위험)
- 시즌 리셋 15% 소각은 장기 공급 안정에 기여
- 분기별 경제 지표 리뷰 사이클 도입 권장

## 3. 게임 디자인

### 3-1. PvP 배틀 시스템

**방식**: 자동 턴제 시뮬레이션 (기존 spec-pvp-battle.md 유지)

- 85% 이탈은 F2P 게임의 정상 수준 (모바일 게임 D1 리텐션 평균 25-35%)

- 목표: Phase 0 유저의 30%가 최소 1회 공유 → 바이럴 퍼널 유지

### 4-1. Viral Measurement System

현재 share_click 이벤트만 추적 중이며, 공유 이후 실제 유입/전환은 측정하지 못함.
바이럴 성장을 주장하려면 전체 퍼널을 측정해야 한다.

**Full Funnel Definition:**
```
Share Click → OG Image Load → Landing Visit (UTM) → Generate Success → Re-share
```

**Per-step tracking method:**

| Funnel Step | Metric | Implementation |
|------------|--------|----------------|
| Share Click | share_twitter, share_farcaster, etc. | Already tracked (events API) |
| OG Image Load | og_image_load | Add counter to OG route (api/og/[address]) |
| Landing Visit | funnel_landing (with UTM source) | Already tracked |
| Generate Success | funnel_generate_success | Already tracked |
| Re-share | share_click with referrer=shared | UTM source matching |

**K-factor formula:**
```
K = (invites sent per user) x (conversion rate per invite)
  = (avg shares per user) x (UTM landing visits / total shares)

Target: K > 0.3 (sustainable growth with marketing)
        K > 1.0 (true viral growth)
```

**Admin Dashboard Requirements:**
- Per-UTM-source funnel: landing -> generate -> share conversion rates
- Daily/weekly K-factor trend
- Top referring sources

**MAU Tracking Plan:**
- Define MAU = unique addresses that completed generation in 30-day window
- Track via server-side events (already partially implemented)
- Replace "MAU 10,000" assumption with scenario-based targets (see above)
- Do NOT claim "viral growth" until K-factor is measured and exceeds 0.3
